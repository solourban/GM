# 매각물건명세서 원문 붙여넣기 후보 추출 설계

상태: 구현 전 설계 확정  
대상: 1단계 원문 붙여넣기 자동추출  
후순위: PDF/OCR 첨부 추출, 법원 문서 자동조회

## 1. 목표

사용자가 매각물건명세서 원문을 붙여넣으면 임차인·점유·특수권리 관련 정보를 **후보로 추출**하고, 사용자가 원문 근거를 확인한 뒤 선택한 값만 Step 2에 반영한다.

이 기능은 문서 내용을 법률적으로 확정하거나 입찰 적합 여부를 판단하지 않는다.

### 추출 대상

- 임차인명
- 점유자명
- 전입일
- 확정일자
- 배당요구일
- 보증금
- 차임
- 점유부분
- 특수권리 의심 문구
- 인수 가능 문구

### 핵심 원칙

1. 자동 확정하지 않는다.
2. 추출만으로 Step 2 입력값이나 권리분석 결과를 변경하지 않는다.
3. 원문과 추출 후보를 같은 검토 화면에서 함께 보여준다.
4. 기존 Step 2 값과 충돌하면 기존 값 유지를 기본값으로 한다.
5. 특수권리·인수 문구는 의심 문구로 표시하며 법률 판단으로 바꾸지 않는다.
6. 1단계 원문은 브라우저 밖으로 보내지 않는다.

## 2. 현재 코드 계약

현재 Step 2와 분석 엔진의 입력 계약은 아래와 같다.

| 영역 | 현재 필드 | 분석 반영 |
| --- | --- | --- |
| 최선순위 권리 | `manual.malso.date/type/holder/amount` | 반영 |
| 임차인 | `manual.tenants[].name/moveIn/fixed/deposit` | 반영 |
| 특수권리 | `manual.specials[].type/holder/date/amount` | 반영 |
| 명세서 확인 정보 | 아직 없음 | 반영하지 않음 |

관련 파일:

- `public/app-v2-core.js`: Step 2 렌더링, 입력 정규화, `/api/analyze` 실행
- `public/app-v2-persist.js`: 사건별 Step 2 입력·분석 결과 저장
- `public/app-v2-case-reset.js`: 사건 전환·현재 사건 초기화
- `public/app-v2-analyze-bridge.js`: 분석 요청에 수동 입력 병합
- `src/analyzer.js`: 실제 권리분석 입력 정규화와 계산

점유자명·배당요구일·차임·점유부분·인수 문구는 기존 분석 필드에 억지로 넣지 않는다. 확인된 값은 Step 2 안의 별도 `명세서 확인 정보` 영역에 보존하고, 현재 분석 계산식에는 전달하지 않는다.

## 3. 사용자 흐름

### 3.1 진입

Step 2 `명세서·권리 입력` 상단에 `매각물건명세서 원문 후보 추출` 영역을 둔다.

초기 상태:

- 원문 붙여넣기 textarea
- `후보 추출` 버튼
- 원문이 서버로 전송되지 않는다는 안내
- 자동 확정하지 않는다는 안내

사건 조회가 완료되어 현재 사건 키가 있을 때만 사용할 수 있다. 현재 사건 키는 `window.__auctionCaseScope.currentCaseKey()`를 우선 사용하며, 사건을 전환하면 해당 사건의 세션 초안만 불러온다.

### 3.2 추출

`후보 추출`을 누르면 다음 순서로 동작한다.

1. 입력 크기와 문자 안전성 검사
2. 브라우저 내 결정론적 파서 실행
3. 원문 근거와 추출 후보를 함께 표시
4. 추출 경고와 확인 필요 항목 표시

이 시점에는 `state.manual`, 권리분석 결과, 저장 후보를 변경하지 않는다.

### 3.3 검토

데스크톱에서는 원문과 후보를 좌우 검토 영역으로, 모바일에서는 원문 다음에 후보를 세로로 표시한다.

각 후보는 아래를 보여준다.

- 추출 필드와 값
- 원문 근거 문장
- 신뢰도: `명시`, `문맥`, `불명확`
- 확인 필요 사유
- Step 2 반영 체크박스

모든 반영 체크박스는 기본 미선택 상태다.

### 3.4 반영

`선택 항목 Step 2 반영`을 누를 때만 값을 반영한다.

- 기존 값이 비어 있으면 선택한 후보를 반영할 수 있다.
- 기존 값과 같은 값이면 중복 추가하지 않는다.
- 기존 값과 다르면 `기존 값 유지`를 기본으로 충돌 선택을 요구한다.
- 사용자가 선택하지 않은 후보는 Step 2에 반영하지 않는다.
- 반영 후에도 현재 세션에서는 원문과 근거를 다시 확인할 수 있다.

## 4. 데이터 계약

### 4.1 미확정 추출 초안

원문과 미확정 후보는 `state.manual`과 분리한다.

```js
{
  version: 1,
  caseKey: "현재 사건 키",
  rawText: "사용자가 붙여넣은 원문",
  rawHash: "원문 변경 감지용 해시",
  parsedAt: "ISO 날짜",
  candidates: {
    occupants: [
      {
        id: "안정적인 후보 ID",
        values: {
          tenantName: "",
          occupantName: "",
          moveIn: "",
          fixed: "",
          claimDate: "",
          deposit: "",
          rent: "",
          occupiedPart: ""
        },
        confidence: {
          tenantName: "explicit",
          moveIn: "context"
        },
        evidence: [
          { start: 0, end: 20, label: "전입일", text: "원문 근거" }
        ],
        issues: []
      }
    ],
    specialRights: [
      {
        id: "후보 ID",
        typeCandidate: "유치권",
        holder: "",
        date: "",
        amount: "",
        phrase: "원문 의심 문구",
        evidence: []
      }
    ],
    takeoverPhrases: [
      {
        id: "후보 ID",
        kind: "possible_takeover",
        phrase: "매수인이 인수할 수 있음",
        evidence: []
      }
    ]
  },
  warnings: []
}
```

미확정 후보의 `confidence`는 표시와 검토 순서에만 사용한다. 신뢰도가 높아도 자동 반영하지 않는다.

### 4.2 확인된 Step 2 정보

기존 분석 필드에 직접 대응하는 확인값만 기존 구조에 반영한다.

| 추출 후보 | Step 2 반영 위치 | 조건 |
| --- | --- | --- |
| 임차인명 | `manual.tenants[].name` | 사용자 확인 |
| 전입일 | `manual.tenants[].moveIn` | 사용자 확인 |
| 확정일자 | `manual.tenants[].fixed` | 사용자 확인 |
| 보증금 | `manual.tenants[].deposit` | 사용자 확인 |
| 명시된 특수권리 | `manual.specials[]` | 허용 유형이며 사용자 확인 |

분석 계산식에 직접 쓰지 않는 확인값은 선택적으로 아래 구조에 보존한다.

```js
manual.specReview = {
  occupants: [
    {
      tenantName: "",
      occupantName: "",
      moveIn: "",
      fixed: "",
      claimDate: "",
      deposit: "",
      rent: "",
      occupiedPart: ""
    }
  ],
  specialRights: [
    { typeCandidate: "", holder: "", date: "", amount: "", note: "" }
  ],
  takeoverNotes: [
    { kind: "possible_takeover", text: "" }
  ]
}
```

`manual.specReview`는 Step 2에서 확인 정보로 보여주지만 `/api/analyze` 입력에는 병합하지 않는다.

### 4.3 저장 범위

| 데이터 | 저장 위치 | 수명 |
| --- | --- | --- |
| 원문 | 사건별 `sessionStorage` | 현재 브라우저 탭 세션 |
| 미확정 후보·근거 | 사건별 `sessionStorage` | 현재 브라우저 탭 세션 |
| 확인된 기존 분석 필드 | 기존 사건별 `localStorage` | 현재 저장 정책 |
| 확인된 `manual.specReview` | 기존 사건별 `localStorage` | 현재 저장 정책 |

원문 초안 키:

```text
auction-note:v2:spec-extraction:<caseKey>
```

보안·용량 원칙:

- 원문은 `localStorage`, 서버 로그, 분석 요청에 저장하지 않는다.
- 원문 근거 범위와 원문 발췌는 `localStorage`에 저장하지 않는다. 사용자가 확인한 요약값과 확인 메모만 저장할 수 있다.
- 원문은 최대 100,000자로 제한한다.
- 세션에는 최근 사건 초안 최대 10개만 유지한다.
- 현재 사건 입력 초기화 시 현재 사건의 원문 초안·미확정 후보·확인된 `specReview`를 삭제한다.
- 다른 사건의 초안과 확인값은 삭제하지 않는다.

`manual.specReview`는 선택 필드로 추가한다. 기존 사건 저장값을 무효화하지 않도록 사건 저장 `SCHEMA_VERSION = 3`을 유지하고, `defaultManual`·`safeManual`에서 누락값을 기본 구조로 보완한다.

## 5. 파싱 규칙

### 5.1 파싱 단계

1. 줄바꿈, 탭, 반복 공백을 정규화하되 원문 위치 매핑을 유지한다.
2. `임차인`, `점유자`, `임대차`, `배당요구`, `비고` 등 구역 표제를 찾는다.
3. 사람 또는 점유 단위로 행을 나눈다.
4. 명시된 라벨 주변에서 날짜·금액·점유부분을 추출한다.
5. 특수권리·인수 가능성 키워드가 포함된 문장을 별도 후보로 모은다.
6. 같은 근거와 같은 값을 중복 제거한다.
7. 모순되거나 불완전한 값을 경고로 남긴다.

### 5.2 날짜

- 지원 예: `2024.01.02`, `2024-01-02`, `2024/01/02`, `20240102`
- 정규화 값과 원문 값을 함께 보존한다.
- 라벨이 없는 날짜를 전입일·확정일자·배당요구일로 임의 지정하지 않는다.
- `미상`, `불명`, `해당없음`은 빈 날짜가 아니라 확인 필요 상태로 표시한다.

### 5.3 금액

- 지원 예: `50,000,000원`, `5천만원`, `보증금 1억`, `월 50만원`
- 보증금과 차임은 라벨을 기준으로 구분한다.
- `없음`, `해당없음`을 숫자 `0`으로 자동 확정하지 않는다.
- 보증금인지 차임인지 불명확한 금액은 반영 후보로 선택할 수 없게 한다.

### 5.4 사람과 점유

- 마스킹된 이름(`김OO`, `주식회사 OOO`)을 원문 그대로 유지한다.
- 임차인과 점유자가 다르면 한 사람으로 합치지 않는다.
- 동일 이름만으로 기존 Step 2 임차인과 자동 병합하지 않는다.
- 동일 이름과 날짜 또는 보증금 중 하나 이상이 일치할 때만 병합 후보로 제시한다.
- 점유부분은 분석용 권리 필드에 넣지 않고 `specReview`에 보존한다.

### 5.5 특수권리·인수 문구

1차 키워드 예:

- `유치권`, `법정지상권`, `분묘기지권`, `예고등기`, `가처분`, `가압류`
- `매수인이 부담`, `매수인에게 인수`, `인수할 수 있음`, `보증금 반환`
- `대항력`, `배당요구`, `점유`, `임대차관계 조사서`

키워드가 포함되어도 위험도·인수 여부를 자동 확정하지 않는다. 문장 전체를 의심 문구로 보여주고 사용자가 확인한다.

`매수인이 인수할 권리 없음`, `인수되는 권리 없음`처럼 부정 표현이 있는 문구는 `no_takeover` 후보로 분리한다. 부정 여부가 불명확하면 `ambiguous`로 표시하며 `possible_takeover`로 올리지 않는다.

기존 Step 2 특수권리 허용 유형과 정확히 일치하는 후보만 사용자 확인 후 `manual.specials[]`에 반영할 수 있다. 나머지는 `specReview.specialRights`에 확인 메모로 보존한다.

## 6. 충돌·중복 처리

### 임차인 후보

- 기존 Step 2 값이 비어 있으면 `빈 값 채우기`로 제안한다.
- 기존 값과 후보가 같으면 `이미 반영됨`으로 표시한다.
- 기존 값과 후보가 다르면 기존 값과 후보를 나란히 보여준다.
- 충돌 상태에서는 `기존 값 유지`가 기본이며 사용자가 교체를 선택해야 한다.
- 이름만 같은 임차인 여러 명은 자동 병합하지 않는다.

### 특수권리 후보

- 기존 특수권리와 유형·권리자·날짜가 모두 같으면 중복으로 처리한다.
- 유형만 같으면 별도 후보로 유지한다.
- 의심 문구만 있고 구조화 필드가 부족하면 분석용 특수권리로 반영할 수 없다.

## 7. 개인정보·보안

- 1단계 파서는 브라우저에서만 실행한다.
- 원문 붙여넣기 동작은 새 서버 API를 만들지 않는다.
- 원문·근거 문구는 `/api/analyze` 요청에 포함하지 않는다.
- 원문을 콘솔, 오류 메시지, 서버 로그에 기록하지 않는다.
- 원문 표시에는 `textContent` 또는 HTML 이스케이프를 사용한다.
- `<script>`, 이벤트 속성 등 붙여넣기 HTML을 실행하지 않는다.
- 100,000자 초과 입력은 파싱하지 않고 사용자에게 줄여 붙여넣도록 안내한다.

| `public/app-v2-core.js` | Step 2 안에 안정적인 추출 영역 mount 지점 제공 |
| `public/app-v2-persist.js` | 확인된 `manual.specReview` 저장·복원 |

| `tests/spec-extractor-regression.test.js` | 자동 확정·서버 전송·사건 혼합 방지 |

### 렌더링

`app-v2-core.js`의 Step 2 렌더링에 안정적인 mount 지점을 둔다. 별도 UI 스크립트는 이 mount 지점 안에서만 렌더링하며 결과 카드 순서나 전체 결과 DOM을 이동하지 않는다.

### 분석 엔진

- 기존 분석 계산식은 변경하지 않는다.
- 확인 전 후보는 분석 입력에 들어가지 않는다.
- `manual.specReview`는 분석 입력에 들어가지 않는다.
- 사용자가 확인해 기존 `manual.tenants` 또는 `manual.specials`에 반영한 값만 현재 분석 계약을 따라 사용한다.

## 9. 구현 순서

### PR 1. 순수 파서와 fixture

- 외부 상태를 변경하지 않는 parser API
- 날짜·금액·라벨·문구 추출
- 비식별 fixture와 단위 테스트
- UI와 Step 2 반영 없음

최소 fixture:

- 명시된 임차인 1명과 날짜·보증금
- 임차인과 점유자가 다른 2개 행
- 배당요구일·차임·점유부분 포함 행
- 특수권리 의심 문구
- `매수인이 인수할 권리 없음` 부정 문구
- 깨진 표와 중복 행
- HTML·스크립트 문자열

### PR 2. 원문·후보 검토 UI

- Step 2 mount 지점
- 사건별 sessionStorage 초안
- 원문과 후보 동시 표시
- 아직 Step 2 반영 없음

### PR 3. 확인 적용과 사건별 저장

- 선택 항목 적용
- 충돌 확인
- `manual.specReview` 저장·복원
- 현재 사건 초기화 연동
- 기존 권리분석 회귀 검증

큰 기능을 한 번에 붙이지 않고 각 PR에서 브라우저 검수를 마친 뒤 다음 단계로 진행한다.

## 10. 검수 기준

1. 원문을 붙여넣고 후보 추출을 눌러도 Step 2 값과 분석 결과가 변하지 않는다.
2. 원문과 각 후보의 근거 문장을 같은 화면에서 확인할 수 있다.
3. 선택하지 않은 후보는 Step 2에 반영되지 않는다.
4. 기존 Step 2 값과 충돌하면 조용히 덮어쓰지 않는다.
5. 점유자명·배당요구일·차임·점유부분은 기존 분석 필드에 잘못 들어가지 않는다.
6. 특수권리·인수 문구는 의심 문구로 표시되고 위험도로 자동 확정되지 않는다.
7. 원문과 미확정 후보는 `/api/analyze`와 새 서버 API로 전송되지 않는다.
8. 현재 사건 초기화 시 현재 사건 초안과 확인값만 삭제된다.
9. 사건 A 초안이 사건 B에서 보이지 않는다.
10. 기존 사건 저장값은 `SCHEMA_VERSION = 3` 유지 상태에서 복원된다.
11. 100,000자 초과·빈 원문·깨진 표·중복 행을 안전하게 처리한다.
12. 원문 내 HTML과 스크립트가 실행되지 않는다.
13. 기존 권리분석·실거래가·외부검증·저장 후보 기능이 깨지지 않는다.
14. 데스크톱과 모바일에서 원문·후보 영역이 겹치거나 페이지를 가로로 밀지 않는다.

## 11. 1단계 제외 범위

- 이미지 첨부와 OCR

- 추출 직후 자동 권리분석 실행
